\begin{abstract}
\noindent
When a transformer attends over long sequences, the bottleneck is rarely arithmetic: it is the cost of writing a full $N{\times}N$ score matrix to GPU global memory and reading it back for softmax. I implemented a FlashAttention-style forward pass from scratch in CUDA C++, replacing that round trip with tiled shared-memory kernels and an online softmax recurrence~\cite{dao2022,milakov2018}. I benchmarked five kernel families on an NVIDIA Tesla T4: a three-kernel naive baseline, tiled FP32 FlashAttention (v1), causal FlashAttention, a warp-centric FP32 experiment (v2), and an FP16 WMMA score path. Tiled Flash cuts latency by roughly $1.9\times$ at $N{=}8192$ and drops global reads by about one order of magnitude at $N{=}4096$. Causal masking gives the best wall-clock time in my runs ($\approx 30.6$\,ms at $N{=}8192$). The WMMA path reaches $\approx 50$\,ms non-causal with the smallest HBM footprint of any variant. The v2 prototype matches the reference but runs slower than even the naive baseline at large $N$, which made for a clear lesson in how tile geometry and memory traffic interact.
\textit{All timings and NCU data in this report come from a Tesla T4 on \textbf{Google Cloud Platform}. I did not use the department Euler cluster for final results: instruction nodes combined CUDA toolchains with host GCC versions that \texttt{nvcc} rejected without brittle workarounds; the GPUs available there were not T4s; and \textbf{Nsight Compute} (\texttt{ncu}) profiling for HBM counters is run with \texttt{sudo} in my workflow (\texttt{run\_ncu\_profile.sh}), which Euler instruction accounts do not allow. I standardized on GCP for reproducibility.}

\section[Problem and goals]{Problem Statement}
\label{sec:problem}

Attention is the engine of modern language models, but the standard implementation has a quiet performance problem. For every forward pass, it writes the full $N{\times}N$ score matrix $S = QK^\top / \sqrt{d}$ to high-bandwidth memory (HBM), then immediately reads it back to compute softmax~\cite{vaswani2017}. At sequence lengths like $N = 4096$ or $N = 8192$, that traffic completely dominates the wall-clock time; the GPU is mostly waiting on memory, not computing.

Dao et al.~\cite{dao2022} showed that you can avoid materializing $S$ at all by tiling $Q$, $K$, and $V$ into blocks that fit in fast on-chip shared memory (SRAM), and tracking softmax statistics incrementally as you go. The result is the same attention output, but with $O(N)$ global memory traffic instead of $O(N^2)$.

I reimplemented that idea from scratch in raw CUDA, without relying on any closed-source kernel libraries. My goals were: (1)~check that my kernels match a reference implementation, (2)~measure how much latency and HBM traffic change on a real T4, and (3)~try a few design variants (causal masking, a warp-centric FP32 layout, and Tensor Core scores) so I could see the tradeoffs between IO, occupancy, and arithmetic directly in the data.

For me this was a performance engineering exercise: I wanted to know \emph{why} a kernel is fast or slow, not just that it is. Attention is the natural place to learn that, and it lines up with the IO-aware and GPU programming themes in ME/CS/ECE 759.

\subsection{Shared utilities (\texttt{attention\_utils.cuh})}

I kept boilerplate in one header instead of duplicating it across every \texttt{.cu} file:
\begin{itemize}
    \item \textbf{\texttt{CUDA\_CHECK}} wraps every runtime call and prints a clear error message with file and line number if something goes wrong.
    \item \textbf{\texttt{GpuTimer}} wraps two \texttt{cudaEvent\_t} handles and returns elapsed milliseconds between \texttt{Start} and \texttt{Stop}.
    \item \textbf{Host helpers:} \texttt{fill\_random} fills $Q$, $K$, $V$ from a fixed seed so every kernel sees the same inputs; \texttt{rmse} computes the root-mean-square error between two output arrays.
    \item \textbf{Tile constants:} \texttt{D\_HEAD=64}; Flash v1 uses \texttt{BR=BC=32}; v2 uses \texttt{BR\_V2=16}, \texttt{BC\_V2=32}; WMMA uses \texttt{BR\_WMMA=BC\_WMMA=16}. All matrices are row-major with shape $[N, d]$.
\end{itemize}

\noindent The header defines \texttt{CUDA\_CHECK} (wraps CUDA calls and exits on failure), \texttt{GpuTimer}, and the compile-time tile sizes every kernel agrees on:

\begin{lstlisting}[language=C++, basicstyle=\ttfamily\footnotesize, frame=single,
  commentstyle=\color{green!60!black}]
// Full CUDA_CHECK macro expands to cudaError_t check + fprintf + exit.
static constexpr int D_HEAD = 64;
static constexpr int BR = 32, BC = 32;       // Flash v1
static constexpr int BR_V2 = 16, BC_V2 = 32; // experimental v2
static constexpr int BR_WMMA = 16, BC_WMMA = 16;
\end{lstlisting}

\subsection{CLI driver (\texttt{main.cu})}

\texttt{main.cu} parses \verb|--mode|, \verb|--target|, \verb|--seq_len|, \verb|--d_head|, \verb|--warmup|, \verb|--iters|, and \verb|--csv|. It allocates and uploads $Q$, $K$, $V$ to the device, then branches into one of two paths:

\begin{itemize}
    \item \textbf{Correctness mode} (\verb|--mode correctness --target X|) allocates an extra $N{\times}N$ buffer \texttt{dS} for the naive score matrix. For non-causal targets, the naive reference runs first into \texttt{dO\_ref}, then the kernel under test runs into \texttt{dO\_tst}. RMSE must be below $10^{-3}$ for FP32 kernels and $5{\cdot}10^{-3}$ for WMMA.
    \item \textbf{Benchmark mode} warms up the kernel, then averages \texttt{iters} timed runs with \texttt{GpuTimer}. The result is printed as plain text or as a CSV line (\texttt{mode,N,d,ms}) with \verb|--csv|.
\end{itemize}

I route kernels through \texttt{run\_target}: \texttt{flash} calls \texttt{flash\_attention\_v1}, \texttt{flash\_causal} calls \texttt{flash\_attention\_v1\_causal}, and the WMMA modes call their wrappers. Naive modes bypass that helper because they still allocate the full $N{\times}N$ \texttt{dS} buffer.

\begin{lstlisting}[language=C++, basicstyle=\ttfamily\footnotesize, frame=single,
  commentstyle=\color{green!60!black}]
static void run_target(const char* mode,
    const float* dQ, const float* dK, const float* dV,
    float* dO, int N, int d)
{
    if      (!strcmp(mode, "flash"))
        flash_attention_v1(dQ, dK, dV, dO, N);
    else if (!strcmp(mode, "flash_causal"))
        flash_attention_v1_causal(dQ, dK, dV, dO, N);
    else if (!strcmp(mode, "flash_v2"))
        flash_attention_v2(dQ, dK, dV, dO, N);
    else if (!strcmp(mode, "flash_wmma"))
        flash_attention_v3_wmma(dQ, dK, dV, dO, N);
    // ... flash_wmma_db, error handling
}
\end{lstlisting}

\subsection{Naive baseline (three kernels)}

The naive path follows the textbook definition directly: compute $S = QK^\top / \sqrt{d}$ and write the entire matrix to HBM, run a row-wise softmax pass over $S$, then compute $O = PV$. The causal variant masks the upper triangle (setting those logits to $-10^{30}$) before softmax so the output matches the lower-triangular attention pattern. That gives me a correct reference and the worst-case HBM traffic story.

\noindent The QKT kernel is a straightforward dot product per $(\text{row},\text{col})$ pair; the causal variant short-circuits upper-triangular entries:

\begin{lstlisting}[language=C++, basicstyle=\ttfamily\footnotesize, frame=single,
  commentstyle=\color{green!60!black}]
__global__ void qkt_kernel(..., float* S, int N, int d) {
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    int col = blockIdx.x * blockDim.x + threadIdx.x;
    if (row >= N || col >= N) return;
    float acc = 0.f;
    for (int k = 0; k < d; ++k)
        acc += Q[row * d + k] * K[col * d + k];
    S[row * N + col] = acc * rsqrtf((float)d);
}

__global__ void qkt_kernel_causal(..., float* S, int N, int d) {
    // ...
    if (col > row) { S[row * N + col] = -1e30f; return; }
    // ... same dot product as dense qkt_kernel
}
\end{lstlisting}

\subsection{FlashAttention v1 (tiled FP32)}

FlashAttention avoids ever storing $S$ in HBM. I use tile sizes $B_r = B_c = 32$ with $d = 64$, so the working $Q$, $K$, and $V$ tiles fit in $\approx 24$\,KB of shared memory. Instead of writing a row of $S$ and reading it back for softmax, I track the running row maximum $m_i$ and normalizer $\ell_i$ in registers and rescale the partial output whenever a new tile raises the maximum. Milakov and Gimelshein~\cite{milakov2018} proved this matches a two-pass softmax numerically. Listing~\ref{lst:online_softmax} shows the rescale I use after each inner tile (from \texttt{flash\_attention\_v1.cu}).

\begin{lstlisting}[language=C++, basicstyle=\ttfamily\footnotesize, frame=single,
  caption={Online softmax rescale and partial $PV$ blend (FlashAttention v1).},
  label=lst:online_softmax, commentstyle=\color{green!60!black}]
float m_new = fmaxf(m_i, m_tilde);
float alpha = expf(m_i - m_new);
float beta  = expf(m_tilde - m_new);
for (int k = 0; k < d; ++k) {
    float pv_k = 0.0f;
    for (int jj = 0; jj < bc_actual; ++jj)
        pv_k += S_row[jj] * V_smem[jj][k];
    O_acc[k] = alpha * O_acc[k] + beta * pv_k;
}
m_i = m_new;
l_i = alpha * l_i + beta * l_tilde;
\end{lstlisting}

\noindent Each iteration rescales the running output by \texttt{alpha} and blends in the new tile's contribution weighted by \texttt{beta}. When the outer loop finishes, the accumulator holds the softmax-weighted output without ever materializing $S$ in global memory.

\subsection{Causal FlashAttention (FP32)}

For autoregressive models, position $i$ should only attend to $j \leq i$. In \texttt{flash\_attention\_v1\_causal.cu} I skip entire $K/V$ column tiles when they lie entirely in the future for the current $Q$ tile. On tiles that cross the diagonal, I mask logits with $j > i$ before the softmax update. For large $N$ that is roughly half the work of the non-causal kernel, which shows up in both latency and NCU reads.

\noindent The early exit in the outer loop over $K/V$ tiles is the main savings: once the start column of a tile is past the last query row in the current $Q$ block, no further tiles can contribute.

\begin{lstlisting}[language=C++, basicstyle=\ttfamily\footnotesize, frame=single,
  caption={Skipping future-only $K/V$ tiles (causal Flash, excerpt).},
  label=lst:causal_skip, commentstyle=\color{green!60!black}]
for (int j = 0; j < Tc; ++j) {
    const int kv_start = j * BC;
    // ...
    if (kv_start > q_row_max_in_block)
        break;   // entire tile is in the future for this Q block
    // load K_smem, V_smem, compute logits, online softmax update ...
}
\end{lstlisting}

\subsection{Experimental v2 kernel (\texttt{flash\_v2})}
\label{sec:flashv2}

I wrote \texttt{flash\_attention\_v2.cu} to try a different thread layout: $B_r = 16$, $B_c = 32$, 512 threads per block, fused score and softmax buffers in shared memory, and \texttt{\_\_launch\_bounds\_\_(512,2)} after an earlier 1024-thread prototype used $\approx 41$\,KB of shared memory. The hope was better occupancy on T4; as the results show, it did not pay off.

\noindent Listing~\ref{lst:v2_kernel} shows the launch hint, warp shuffle helper, and shared buffers for scores plus running softmax state (one warp per query row within the $B_r$ rows of the $Q$ tile).

\begin{lstlisting}[language=C++, basicstyle=\ttfamily\footnotesize, frame=single,
  caption={\texttt{flash\_attn\_fwd\_v2}: occupancy hint, warp reduce, shared tiles.},
  label=lst:v2_kernel, commentstyle=\color{green!60!black}]
static __device__ inline float warp_reduce_sum(float v) {
    for (int offset = 16; offset > 0; offset >>= 1)
        v += __shfl_down_sync(0xffffffffu, v, offset);
    return v;
}

template <int BR, int BC, int D>
__global__ void __launch_bounds__(512, 2)
flash_attn_fwd_v2(const float* Q, const float* K, const float* V,
                  float* O, int N)
{
    __shared__ float Q_smem[BR][D];
    __shared__ float K_smem[BC][D], V_smem[BC][D];
    __shared__ float tile_scr[BR][BC];  // logits, then softmax weights
    // ... online softmax + PV accumulation ...
}
\end{lstlisting}

\subsection{WMMA scores (Tensor Cores)}

\texttt{flash\_attention\_v3\_wmma.cu} calls WMMA (\texttt{m16n16k16}) for $16{\times}16$ score tiles with FP16 operands on the T4's Tensor Cores, while I keep softmax state and the output in FP32~\cite{nvidia_wmma}. The \texttt{flash\_wmma\_db} entry point is the same kernel on sm\_75; a real async double-buffer path would need sm\_80 or newer.

\noindent Listing~\ref{lst:wmma_snippet} is the inner part of the score computation: I convert $Q$ and $K$ tiles to half precision in shared memory, fill WMMA fragments, and accumulate the $16{\times}16$ tile of logits into \texttt{Stile} before softmax.

\begin{lstlisting}[language=C++, basicstyle=\ttfamily\footnotesize, frame=single,
  caption={WMMA $Q K^\top$ tile (FP16 matmul, FP32 accumulator).},
  label=lst:wmma_snippet, commentstyle=\color{green!60!black}]
wmma::fragment<wmma::matrix_a, 16, 16, 16, half, wmma::row_major> a_frag;
wmma::fragment<wmma::matrix_b, 16, 16, 16, half, wmma::col_major> b_frag;
wmma::fragment<wmma::accumulator, 16, 16, 16, float> c_frag;
wmma::fill_fragment(c_frag, 0.0f);
for (int kk = 0; kk < D; kk += 16) {
    wmma::load_matrix_sync(a_frag, &Qh[0][kk], D);
    // pack B from Kh into Bpack, then:
    wmma::load_matrix_sync(b_frag, Bpack, 16);
    wmma::mma_sync(c_frag, a_frag, b_frag, c_frag);
}
wmma::store_matrix_sync(&Stile[0][0], c_frag, BC, wmma::mem_row_major);
\end{lstlisting}

The traffic tables~\ref{tab:hbm4096} and~\ref{tab:hbm8192} match what I saw on the timer. At $N = 4096$, naive reads $\approx 25.8$\,GB and writes $\approx 1.14$\,GB; \texttt{flash} reads $\approx 2.21$\,GB and writes only $8.4$\,MB. That is about $12\times$ fewer reads and orders of magnitude fewer writes, because I stopped shuttling the full $S$ matrix through HBM. Writes tell the same story: naive must dump the $N{\times}N$ scores; every tiled variant only writes $O$, so they all land near the same $\approx 8.4$\,MB.

\texttt{flash\_causal} reads about half as much as \texttt{flash} at $N = 8192$ (4.4\,GB vs 8.8\,GB), which directly reflects the tile-skipping: for a given $Q$ tile, all $K/V$ tiles to its right are simply never loaded.

\texttt{flash\_v2} is the failure I learned the most from. It reads $\approx 17.6$\,GB at $N = 8192$, roughly twice \texttt{flash}, even though I still fused softmax in SRAM. I chose $B_r = 16$ instead of $32$, so the outer loop over $Q$ runs twice as many times; each pass still streams all $N / B_c$ tiles of $K/V$. That doubles $K/V$ traffic versus v1 and explains why the kernel falls behind naive despite correct outputs. It drove home that IO efficiency means matching tile shapes to reuse, not only ``fusing'' softmax.

WMMA reaches the lowest read volume at $N = 8192$ ($\approx 2.2$\,GB), below even \texttt{flash\_causal}. FP16 operands are smaller than FP32, so the same $16{\times}16$ score tile loads fewer bytes through L1TEX. Combined with Tensor Core throughput for the $QK^\top$ step, WMMA improves on \texttt{flash} in both traffic and runtime.

The roofline panel (Figure~\ref{fig:roofline}) is only a rough diagnostic: I used the same $4N^2d$ FLOP estimate for every kernel regardless of precision, and NCU bytes do not line up with an ideal roofline. Still, it separates the clearly memory-heavy kernels (naive, \texttt{flash\_v2}) from the leaner ones (WMMA, causal Flash), which matches the latency ordering I saw.

\section[Takeaways]{Conclusions and Future Work}
\label{sec:conclusions}

To rebuild the binary, rerun benchmarks and NCU, and regenerate figures from the CSVs, follow \texttt{FinalProject/README.md} (same steps as Section~\ref{sec:deliverables}, with paths relative to \texttt{FinalProject/}).

The clearest lesson for me is that tiling alone does not win: you have to match tile shapes to how often you reload $K$ and $V$. The naive baseline shows $O(N^2)$ HBM traffic crushing runtime at large $N$. My FlashAttention v1 kernel fixes that by never writing the full score matrix to HBM, and the $\approx 1.9\times$ speedup at $N = 8192$ lines up with the traffic drop in NCU. Causal masking goes further by skipping work the dense problem never needed, which gave the best latency in my sweep. WMMA cuts reads even further and speeds up the non-causal path, but I did not fuse causal skipping into it yet, so it still loses to causal FP32 on wall-clock time.

In \texttt{flash\_v2} NCU shows twice the reads of v1 because halving $B_r$ doubles how often I sweep $K/V$. That is easy to overlook when you are tuning occupancy. It reminded me that the bottleneck is often memory access patterns, not whether softmax is ``fused'' in principle.

Next, I would fuse causal masking into the WMMA path, revisit \texttt{flash\_v2} with v1-sized tiles for $Q$, and maybe explore KV-cache-aware inference. Longer term, multi-GPU attention and FlashAttention-2/3-style scheduling~\cite{dao2022} are on my list.
